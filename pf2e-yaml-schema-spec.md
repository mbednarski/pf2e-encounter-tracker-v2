# PF2e Encounter Tracker v2 - YAML Schema Reference

**Version:** 0.1 (draft)
**Date:** 2026-04-27
**Status:** Ready for review

---

## 1. Purpose

YAML is the human-editable import/export format for library data and encounter snapshots. The YAML schema mirrors the canonical TypeScript types from the subsystem specs; there is no separate hand-maintained schema language.

This document locks the document kinds, round-trip rules, validation expectations, and import/export boundaries so implementation can start without inventing new file formats.

---

## 2. Document Envelope

Every YAML file uses a small envelope:

```yaml
kind: creature
schemaVersion: 1
data:
  id: goblin-warrior
  name: Goblin Warrior
```

```typescript
type YamlDocumentKind =
  | "creature"
  | "spell"
  | "effect-definition"
  | "hazard"
  | "party-member"
  | "companion"
  | "party"
  | "encounter"

interface YamlDocument<T> {
  kind: YamlDocumentKind
  schemaVersion: 1
  data: T
}
```

`schemaVersion` is always `1` for V2. Migration code may be stubbed, but no migration behavior is required yet.

---

## 3. Data Shapes

The `data` payload is a direct serialization of the canonical TypeScript type:

| Kind | Canonical Type | Source Spec |
|---|---|---|
| `creature` | `Creature` | Creature types spec |
| `spell` | `Spell` | Creature types spec §5.2 |
| `effect-definition` | `EffectDefinition` | Effects & durations spec |
| `hazard` | `Hazard` | Hazards spec |
| `party-member` | `PartyMember` | Party members spec |
| `companion` | `Companion` | Party members spec |
| `party` | `Party` | Party members spec |
| `encounter` | `EncounterState` | Architecture spec |

YAML keys use the same camelCase property names as the TypeScript interfaces. IDs are stable strings supplied by importers or generated by the UI before export.

---

## 4. Import Rules

- Unknown `kind` is rejected.
- Unknown `schemaVersion` is rejected.
- Required fields from the canonical TypeScript type must be present.
- Optional fields may be omitted.
- Unknown extra fields are preserved only if the importer explicitly stores raw extension data; V2 default is to reject them as validation issues.
- YAML import produces parsed data plus validation issues. It does not mutate IndexedDB directly; the orchestrator decides whether to save.
- Multi-document YAML streams are allowed. Each document is validated independently.

The LLM parser may emit `creature` and `effect-definition` documents together when a statblock includes afflictions.

---

## 5. Export Rules

- Export writes the same envelope shape used for import.
- Round-tripping a valid V2 document through import and export should preserve all modeled fields.
- Runtime-only UI state is not exported unless it is part of `EncounterState`.
- API keys and settings are never exported.
- Undo history is never exported.

---

## 6. Test Priority

1. **Envelope validation** - reject unknown kind, missing schemaVersion, and unsupported schemaVersion.
2. **Creature round-trip** - YAML creature with attacks, abilities, spellcasting, and traits imports and exports without losing modeled fields.
3. **Effect definition round-trip** - condition, persistent damage, and affliction definitions preserve suggestions, modifiers, traits, and affliction data.
4. **Multi-document import** - creature plus affliction effect definitions returns all valid documents and per-document validation issues.
5. **Sensitive data exclusion** - settings and API keys never appear in exported YAML.
