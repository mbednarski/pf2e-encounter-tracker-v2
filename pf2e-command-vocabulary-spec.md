# PF2e Encounter Tracker v2 — Command Vocabulary Specification

**Version:** 0.1 (draft)
**Date:** 2026-04-22
**Status:** Ready for review

---

## 1. Design Decisions

### 1.1 Command Structure

Commands are minimal serializable objects: `id`, `type`, and a type-specific `payload`. No timestamp, no source metadata. The orchestrator stamps combat log entries with contextual metadata at dispatch time — the domain doesn't care who sent a command or when.

### 1.2 Validation Policy

The domain never throws. Invalid commands return `{ newState: state /* unchanged */, events: [{ type: "command-rejected", ... }] }`. The orchestrator surfaces rejected commands in the combat log and does **not** push them onto the undo stack (no state change → nothing to undo).

### 1.3 HP Commands

`APPLY_DAMAGE` and `APPLY_HEALING` take final numbers. The GM resolves resistances, weaknesses, and immunities mentally. An optional `damageType` field is cosmetic — used only in combat log entries.

### 1.4 Effect Commands

`APPLY_EFFECT` takes the minimum required payload. The domain fills defaults from the `EffectDefinition`: duration defaults to `unlimited`, value defaults to 1 for value conditions, implied effects are auto-created. All defaults are overridable.

## 3. Phase Restrictions

Each command is legal in specific encounter phases. Dispatching a command in an illegal phase produces a `command-rejected` event.

| REMOVE_EFFECT | ✓ | ✓ | ✓ | — |

| MODIFY_EFFECT_VALUE | ✓ | ✓ | ✓ | — |
| RESOLVE_PROMPT | — | — | ✓ | — |
| MARK_REACTION_USED | — | ✓ | ✓ | — |

| SET_NOTE | ✓ | ✓ | ✓ | — |
| MARK_DEAD | ✓ | ✓ | ✓ | — |
| REVIVE | ✓ | ✓ | ✓ | — |

**Notes on the table:**

- HP and effect commands are allowed in RESOLVING because prompt resolution may require applying damage (persistent damage) or modifying effects. The GM should be able to fix things mid-resolution without being blocked.
- RENAME_COMBATANT is allowed in RESOLVING — it's harmless and the GM might realize a naming error while looking at prompts.
- Nothing is allowed in COMPLETED except RESET_ENCOUNTER. The encounter is read-only.
- ADD_COMBATANT in ACTIVE supports creatures joining mid-combat (reinforcements). The new combatant is added to state but not to initiative — the GM uses REORDER_COMBATANT or SET_INITIATIVE_ORDER to place them.

## 4. Command Definitions

### 4.1 Encounter Lifecycle

#### START_ENCOUNTER

Transitions from PREPARING → ACTIVE. Begins round 1.

```typescript
interface StartEncounterPayload {
  // no fields — all setup is done via other commands during PREPARING
}
```

**Validation:**
- Phase must be PREPARING
- At least 2 combatants must exist
- Initiative order must be set (non-empty `initiative.order`)

**State changes:**
- `phase → ACTIVE`
- `round → 1`
- `initiative.currentIndex → 0`

**Events:**
- `{ type: "encounter-started" }`
- `{ type: "turn-started", combatantId: firstCombatant, round: 1 }`
- Start-of-turn prompt generation for first combatant (if any effects have `turnStartSuggestion`)

#### COMPLETE_ENCOUNTER

Transitions from ACTIVE → COMPLETED. Combat is over.

```typescript
interface CompleteEncounterPayload {
  // no fields
}
```

**Validation:**
- Phase must be ACTIVE

**State changes:**
- `phase → COMPLETED`

**Events:**
- `{ type: "encounter-completed" }`

#### RESET_ENCOUNTER

Transitions from COMPLETED → PREPARING. Clears all encounter-specific state for re-run.

```typescript
interface ResetEncounterPayload {
  // no fields
}
```

**Validation:**
- Phase must be COMPLETED

**State changes:**
- `phase → PREPARING`
- `round → 0`
- `initiative → { order: [], currentIndex: -1, delaying: [] }`
- All combatants: `currentHp → baseStats.hp`, `tempHp → 0`, `appliedEffects → []`, `reactionUsedThisRound → false`, `isAlive → true`
- `pendingPrompts → []`
- `combatLog → []`

**Events:**
- `{ type: "encounter-reset" }`

### 4.2 Combatant Management

#### ADD_COMBATANT

Adds a combatant to the encounter. The combatant is a deep clone of a creature template, optionally with weak/elite adjustment. The domain receives the fully constructed `CombatantState` — cloning and template application happen in the orchestrator (or a factory function the orchestrator calls).

```typescript
interface AddCombatantPayload {
  combatant: CombatantState     // fully constructed, ready to insert
}
```

**Why the domain receives a complete CombatantState:** The domain is a state machine — it applies commands to state. Constructing a combatant from a creature template + weak/elite adjustments is a *creation* concern, not a *state transition* concern. A factory function in `domain/creatures/` handles construction; the orchestrator calls it, then dispatches ADD_COMBATANT with the result. This keeps the reducer simple: insert the combatant, done.

**Validation:**
- Phase must be PREPARING or ACTIVE
- `combatant.id` must not already exist in state

**State changes:**
- Insert combatant into `combatants` record

**Events:**
- `{ type: "combatant-added", combatantId, name }`

**Note:** The combatant is NOT added to initiative order. In PREPARING, the GM sets the full order with SET_INITIATIVE_ORDER. In ACTIVE (mid-combat join), the GM explicitly places them with REORDER_COMBATANT.

#### REMOVE_COMBATANT

Removes a combatant entirely from the encounter. Destructive — use MARK_DEAD for creatures that died but should remain visible.

```typescript
interface RemoveCombatantPayload {
  combatantId: CombatantId
}
```

**Validation:**
- Phase must be PREPARING or ACTIVE
- Combatant must exist
- Cannot remove the combatant whose turn it currently is (in ACTIVE phase)

**State changes:**
- Remove from `combatants` record
- Remove from `initiative.order` (if present)
- Remove from `initiative.delaying` (if present)
- Remove any `AppliedEffect` instances on other combatants where `sourceId` is this combatant
- If removal shifts `initiative.currentIndex`, adjust pointer to keep the same combatant's turn active

**Events:**
- `{ type: "combatant-removed", combatantId, name }`
- Any effect-removed events from orphaned effects

#### RENAME_COMBATANT

```typescript
interface RenameCombatantPayload {
  combatantId: CombatantId
  newName: string
}
```

**Validation:**
- Combatant must exist

**State changes:**
- `combatants[id].name → newName`

**Events:**
- `{ type: "combatant-renamed", combatantId, oldName, newName }`

### 4.3 Initiative

#### SET_INITIATIVE_ORDER

Sets the complete initiative order. Replaces any existing order. Used during PREPARING to establish initial order, or during ACTIVE to do a full reorder.

```typescript
interface SetInitiativeOrderPayload {
  order: CombatantId[]
}
```

**Validation:**
- Phase must be PREPARING or ACTIVE
- All IDs in `order` must reference existing, alive combatants
- No duplicates in `order`

**State changes:**
- `initiative.order → order`
- In ACTIVE: `initiative.currentIndex` is adjusted to point to the same combatant that was active before the reorder (by ID lookup, not position). If the active combatant is no longer in the new order, index resets to 0.

**Events:**
- `{ type: "initiative-set", order }`

#### REORDER_COMBATANT

Moves a single combatant to a new position in the initiative order. Also used to insert a combatant into initiative (from not-in-order or from delaying).

```typescript
interface ReorderCombatantPayload {
  combatantId: CombatantId
  newIndex: number              // position in order (0-based)
}
```

**Validation:**
- Phase must be PREPARING or ACTIVE
- Combatant must exist and be alive
- `newIndex` must be in range [0, order.length] (can append to end)

**State changes:**
- Remove combatant from current position in `initiative.order` (if present)
- Remove from `initiative.delaying` (if present)
- Insert at `newIndex`
- Adjust `initiative.currentIndex` to keep the active combatant stable

**Events:**
- `{ type: "initiative-changed", combatantId, newIndex }`

#### END_TURN

Signals that the current combatant's turn is ending. Triggers the blocking resolution flow.

```typescript
interface EndTurnPayload {
  // no fields — always applies to the current combatant
}
```

**Validation:**
- Phase must be ACTIVE
- `initiative.currentIndex` must point to a valid combatant

**State changes:**
- Emit `turn-ended` event for current combatant
- Process hard clock expirations: remove all effects on the current combatant (and effects on OTHER combatants) with `duration: { type: "untilTurnEnd", combatantId: currentCombatant }`
- Generate `pendingPrompts` from `turnEndSuggestion` on remaining effects
- If prompts exist: `phase → RESOLVING`
- If no prompts: advance immediately (see §5 Turn Flow)

**Events:**
- `{ type: "turn-ended", combatantId }`
- Any `effect-removed` events from hard clock expirations
- Any `prompt-generated` events (informational, for combat log)

#### DELAY

Current combatant delays their turn.

```typescript
interface DelayPayload {
  // no fields — always applies to the current combatant
}
```

**Validation:**
- Phase must be ACTIVE
- Current combatant must not have already acted this turn (not enforceable by domain — this is a GM judgment call, so no validation here)

**State changes:**
- Remove current combatant from `initiative.order`
- Add to `initiative.delaying`
- Advance `initiative.currentIndex` to next combatant (wrapping and incrementing round if needed)
- Emit turn-started for the new current combatant

**Events:**
- `{ type: "combatant-delayed", combatantId }`
- `{ type: "turn-started", combatantId: nextCombatant, round }`
- Start-of-turn effects processing for next combatant

#### RESUME_FROM_DELAY

A delaying combatant re-enters initiative at a specified position and immediately begins their turn.

```typescript
interface ResumeFromDelayPayload {
  combatantId: CombatantId
  insertIndex: number           // position in initiative.order
}
```

**Validation:**
- Phase must be ACTIVE
- Combatant must be in `initiative.delaying`
- `insertIndex` must be in range [0, order.length]
- Cannot resume during another combatant's RESOLVING phase

**State changes:**
- END_TURN must be dispatched for the currently active combatant first (their turn is interrupted by the delaying combatant's re-entry). The orchestrator handles this — RESUME_FROM_DELAY itself assumes the current turn has been properly ended.
- Remove from `initiative.delaying`
- Insert into `initiative.order` at `insertIndex`
- Set `initiative.currentIndex → insertIndex`
- Auto-reset reaction for the resuming combatant
- Process start-of-turn hard clock expirations
- Generate start-of-turn prompts (if any → phase → RESOLVING)

**Events:**
- `{ type: "combatant-resumed-from-delay", combatantId, insertIndex }`
- `{ type: "turn-started", combatantId, round }`
- Start-of-turn effect events (if any)

### 4.4 HP & Damage

#### APPLY_DAMAGE

```typescript
interface ApplyDamagePayload {
  combatantId: CombatantId
  amount: number                // positive integer, final value after GM resolves resistances
  damageType?: string           // cosmetic — for combat log ("fire", "slashing", etc.)
}
```

**Validation:**
- Combatant must exist
- `amount` must be > 0

**State changes:**
- Reduce `tempHp` first, then `currentHp`
  - If `amount <= tempHp`: `tempHp -= amount`
  - If `amount > tempHp`: `currentHp -= (amount - tempHp)`, `tempHp → 0`
- `currentHp` floors at 0 (never negative)
- If `currentHp` reaches 0 and combatant `isAlive`: the domain does NOT auto-apply Dying or mark dead. It emits an `hp-reached-zero` event. The GM decides what happens (Dying for PCs, death for NPCs, or narrative override). This is consistent with GM authority.

**Events:**
- `{ type: "hp-changed", combatantId, from, to, tempHpFrom, tempHpTo, cause: "damage", damageType? }`
- `{ type: "hp-reached-zero", combatantId }` (if applicable)

#### APPLY_HEALING

```typescript
interface ApplyHealingPayload {
  combatantId: CombatantId
  amount: number                // positive integer
}
```

**Validation:**
- Combatant must exist
- `amount` must be > 0

**State changes:**
- `currentHp = min(currentHp + amount, baseStats.hp)` — capped at max HP

**Events:**
- `{ type: "hp-changed", combatantId, from, to, cause: "healing" }`

#### SET_TEMP_HP

```typescript
interface SetTempHpPayload {
  combatantId: CombatantId
  amount: number                // non-negative integer
}
```

**Validation:**
- Combatant must exist
- `amount` must be >= 0

**State changes:**
- `tempHp → amount`
- PF2e rule: temp HP doesn't stack, you take the higher value. But this is a SET — the GM decides. If they want to replace lower with higher, they set. If they want to keep the old value, they don't dispatch.

**Events:**
- `{ type: "temp-hp-changed", combatantId, from, to }`

#### SET_HP

Direct HP override. For corrections, GM fiat, or edge cases.

```typescript
interface SetHpPayload {
  combatantId: CombatantId
  amount: number                // non-negative integer
}
```

**Validation:**
- Combatant must exist
- `amount` must be >= 0

**State changes:**
- `currentHp → amount` (no cap against max — GM override is absolute)
- If `amount` is 0 and was previously > 0: emit `hp-reached-zero`
- If `amount` is > 0 and was previously 0: no special behavior (GM handles Dying removal separately)

**Events:**
- `{ type: "hp-changed", combatantId, from, to, cause: "set" }`
- `{ type: "hp-reached-zero", combatantId }` (if applicable)

### 4.5 Effects

#### APPLY_EFFECT

Applies an effect to a combatant. The domain looks up the `EffectDefinition` from the effect library, fills defaults, and creates implied effects.

```typescript
interface ApplyEffectPayload {
  effectId: string              // references EffectDefinition
  targetId: CombatantId
  sourceId: CombatantId         // who caused this effect
  value?: number                // for value conditions; defaults to 1 if hasValue
  duration?: Duration           // defaults to { type: "unlimited" }
  note?: string                 // free text (e.g., "2d6" for persistent damage)
}
```

**Validation:**
- Target combatant must exist
- Source combatant must exist (can be same as target)
- `effectId` must resolve in the effect library
- If effect `hasValue` and `value` is provided: must be >= 1 and <= `maxValue`
- If effect `!hasValue` and `value` is provided: rejected

**State changes:**
- Generate unique `instanceId`
- Create `AppliedEffect` with resolved defaults
- Push onto `combatants[targetId].appliedEffects`
- For each entry in `impliedEffects`: recursively apply (same source, same target, `parentInstanceId` set to the new instance, value 1 if applicable, duration unlimited)
  - **Duplicate check:** If the target already has an implied effect from a different source (no parent, or different parent), do NOT create a duplicate. Each effect source gets its own instance. This means a creature can have Off-Guard from Prone (implied) AND Off-Guard from a spell (separate instance). Both exist; removing Prone removes only its implied Off-Guard.
- **Death subsystem:** If `effectId` is `"dying"`, invoke the death tracking subsystem (conditions library spec §3.2) instead of standard application. The value is adjusted by Wounded, and the death threshold is checked.

**Events:**
- `{ type: "effect-applied", combatantId: targetId, effectName, value?, instanceId }`
- Events for each implied effect created
- Death subsystem events if applicable

#### REMOVE_EFFECT

Removes a specific effect instance from a combatant.

```typescript
interface RemoveEffectPayload {
  targetId: CombatantId
  instanceId: string            // the specific AppliedEffect instance to remove
}
```

**Validation:**
- Target combatant must exist
- Instance must exist in target's `appliedEffects`

**State changes:**
- Remove the instance from `appliedEffects`
- **Cascade:** Remove all instances where `parentInstanceId` matches this instance's `instanceId` (implied effect cleanup)
- **Death subsystem:** If `effectId` is `"dying"`, invoke recovery logic (conditions library spec §3.4): increment or apply Wounded.

**Events:**
- `{ type: "effect-removed", combatantId: targetId, effectName, reason: "removed" }`
- Events for each cascaded implied effect removal
- Death subsystem events if applicable

#### SET_EFFECT_VALUE

Sets the value of a value-based effect to an absolute number.

```typescript
interface SetEffectValuePayload {
  targetId: CombatantId
  instanceId: string
  newValue: number
}
```

**Validation:**
- Target and instance must exist
- Effect must have `hasValue: true`
- `newValue` must be >= 1 and <= `maxValue`. Setting to 0 is not allowed — use REMOVE_EFFECT instead.

**State changes:**
- `appliedEffects[instance].value → newValue`
- **Death subsystem:** If `effectId` is `"dying"`, invoke threshold check (conditions library spec §3.3/§3.5)

**Events:**
- `{ type: "effect-value-changed", combatantId: targetId, effectName, from, to }`

#### MODIFY_EFFECT_VALUE

Relative adjustment. "+1" or "-1" to the current value. Convenience command for prompt resolution.

```typescript
interface ModifyEffectValuePayload {
  targetId: CombatantId
  instanceId: string
  delta: number                 // +N or -N
}
```

**Validation:**
- Target and instance must exist
- Effect must have `hasValue: true`
- Resulting value must be <= `maxValue`
- If resulting value exceeds `maxValue`: rejected

**State changes:**
- `appliedEffects[instance].value += delta`
- **Auto-removal:** If resulting value <= 0, treat as removal — trigger REMOVE_EFFECT logic including implied effect cascades and death subsystem recovery. This is the primary path for prompt-driven removal (e.g., Frightened 1 decremented by 1 → value 0 → removed).
- **Death subsystem:** If `effectId` is `"dying"` and delta is positive, invoke threshold check on increase

**Events:**
- `{ type: "effect-value-changed", combatantId: targetId, effectName, from, to }`

### 4.6 Turn Resolution

#### RESOLVE_PROMPT

The GM's response to a pending prompt during the RESOLVING phase.

```typescript
interface ResolvePromptPayload {
  promptId: string
  resolution: PromptResolution
}

type PromptResolution =
  | { type: "accept" }                  // accept the suggestion
  | { type: "setValue"; value: number } // accept but override the value
  | { type: "dismiss" }                 // keep effect as-is, skip this prompt
  | { type: "remove" }                  // remove the effect entirely
```

For complex multi-outcome prompts (Dying recovery check), the UI presents labeled outcome buttons (Crit Success / Success / Failure / Crit Failure), each of which dispatches a RESOLVE_PROMPT with the appropriate resolution. For example, a Dying 2 recovery check:
- Crit Success → `{ type: "setValue", value: 0 }` → triggers auto-removal (value ≤ 0)
- Success → `{ type: "setValue", value: 1 }`
- Failure → `{ type: "setValue", value: 3 }`
- Crit Failure → `{ type: "setValue", value: 4 }` → triggers death threshold check

**Validation:**
- Phase must be RESOLVING
- `promptId` must exist in `pendingPrompts`
- If `setValue`: value must be >= 0 (value 0 triggers removal)

**State changes:**
- Resolve based on the prompt's `suggestionType` and the chosen `resolution`:
  - `suggestDecrement` + `accept` → decrease effect value by suggested amount. If value reaches 0, remove.
  - `suggestDecrement` + `setValue` → set effect to specified value. If value ≤ 0, remove.
  - `suggestDecrement` + `dismiss` → no change
  - `suggestDecrement` + `remove` → remove effect
  - `confirmSustained` + `accept` → keep effect (no change)
  - `confirmSustained` + `dismiss` → keep effect (same as accept for sustained)
  - `confirmSustained` + `remove` → remove effect (caster didn't sustain)
  - `promptResolution` + `accept` → execute the "positive outcome" (remove persistent damage, etc.)
  - `promptResolution` + `setValue` → set effect to specified value
  - `promptResolution` + `dismiss` → keep as-is (failed the check, effect stays)
  - `promptResolution` + `remove` → remove effect
  - `reminder` + any → remove the prompt, no state change. Reminders are informational.
- **Death subsystem:** When resolving a prompt for a `dying` effect, all setValue and remove operations invoke the death subsystem (threshold checks on increase, Wounded logic on removal). The RESOLVE_PROMPT reducer delegates to the same death tracking code as MODIFY_EFFECT_VALUE and REMOVE_EFFECT.
- Remove the resolved prompt from `pendingPrompts`
- If `pendingPrompts` is now empty:
  - Advance initiative to next combatant
  - Emit `turn-started` for next combatant
  - Process start-of-turn hard clock expirations (`untilTurnStart` effects)
  - Generate start-of-turn prompts
  - If start-of-turn prompts exist: stay in RESOLVING
  - If no prompts: `phase → ACTIVE`

**Events:**
- Effect-related events matching the resolution (effect-removed, effect-value-changed, etc.)
- `{ type: "prompt-resolved", promptId, resolution }`
- Turn advancement events if this was the last prompt

### 4.7 Combat State

#### MARK_REACTION_USED

```typescript
interface MarkReactionUsedPayload {
  combatantId: CombatantId
}
```

**Validation:**
- Combatant must exist and be alive

**State changes:**
- `reactionUsedThisRound → true`

**Events:**
- `{ type: "reaction-used", combatantId }`

#### RESET_REACTION

```typescript
interface ResetReactionPayload {
  combatantId: CombatantId
}
```

**Validation:**
- Combatant must exist

**State changes:**
- `reactionUsedThisRound → false`

**Events:**
- `{ type: "reaction-reset", combatantId }`

**Note on automatic reaction reset:** Reactions reset at the start of a combatant's turn in PF2e. The domain should automatically reset `reactionUsedThisRound` when `turn-started` fires for a combatant. This is unambiguous mechanical behavior — same justification as hard clock expiry. RESET_REACTION exists as a manual override (GM realizes they tracked it wrong).

#### SET_NOTE

```typescript
interface SetNotePayload {
  combatantId: CombatantId
  note: string | null           // null clears the note
}
```

**Validation:**
- Combatant must exist

**State changes:**
- `combatants[id].notes → note`

**Events:**
- `{ type: "note-changed", combatantId }`

#### MARK_DEAD

Marks a combatant as dead. They remain in initiative order (greyed out, skipped during turn advancement) and keep their effects for reference.

```typescript
interface MarkDeadPayload {
  combatantId: CombatantId
}
```

**Validation:**
- Combatant must exist
- Combatant must be alive

**State changes:**
- `isAlive → false`
- If the dead combatant is the current turn holder: advance to next combatant (emit turn-ended, then turn-started for next)

**Events:**
- `{ type: "combatant-died", combatantId }`
- Turn advancement events if applicable

#### REVIVE

Reverses MARK_DEAD. For corrections or game effects (e.g., Breath of Life).

```typescript
interface RevivePayload {
  combatantId: CombatantId
}
```

**Validation:**
- Combatant must exist
- Combatant must be dead (`isAlive === false`)

**State changes:**
- `isAlive → true`

**Events:**
- `{ type: "combatant-revived", combatantId }`

## 5. Turn Flow — Detailed Sequence

The arch spec defined both ADVANCE_TURN and END_TURN. On review, these collapse into a single command: **END_TURN**. There is no scenario where the GM wants to advance initiative without processing turn-end effects — that would skip prompts, which violates the blocking resolution design.

### 5.1 The Single Turn Advancement Path

```
GM presses "End Turn" (or equivalent)
  → dispatch END_TURN
    → domain processes:
        1. Emit turn-ended for current combatant
        2. Hard clock expiry: remove effects with untilTurnEnd matching current combatant
        3. Collect turn-end suggestions from remaining effects
        4. If suggestions exist → generate prompts, phase → RESOLVING
        5. If no suggestions → advance immediately (step 6)
  → [RESOLVING phase — GM resolves prompts one by one via RESOLVE_PROMPT]
  → When last prompt resolved:
        6. Advance initiative.currentIndex to next alive combatant
           - Skip dead combatants
           - Wrap around to 0 and increment round when passing end
        7. Auto-reset reaction for the new current combatant
        8. Emit turn-started for new current combatant
        9. Hard clock expiry: remove effects with untilTurnStart matching new combatant
       10. Collect turn-start suggestions from effects on new combatant
       11. If suggestions exist → generate prompts, stay in RESOLVING
       12. If no suggestions → phase → ACTIVE
```

### 5.2 Round Boundary

When `initiative.currentIndex` wraps past the end of `order`:
- `round += 1`
- `currentIndex → 0` (first alive combatant)
- Emit `{ type: "round-started", round }`

### 5.3 All-Dead Edge Case

If all combatants are dead (or only dead combatants remain in initiative), turn advancement has nowhere to go. The domain emits `{ type: "all-combatants-dead" }` and the orchestrator can suggest COMPLETE_ENCOUNTER to the GM.

## 7. Prompt Data Structure

Prompts are generated by the domain and placed in `pendingPrompts` during turn boundary processing.

```typescript
interface Prompt {
  id: string                      // unique, generated by domain
  combatantId: CombatantId        // whose turn this relates to
  effectInstanceId: string        // the effect that generated this prompt
  effectName: string              // display name
  boundary: "start" | "end"       // which turn boundary generated this
  suggestionType: TurnBoundarySuggestion  // the suggestion from EffectDefinition
  currentValue?: number           // current effect value (for display)
  suggestedValue?: number         // what the value would become if accepted
}
```

The UI renders prompts based on `suggestionType`:

- `suggestDecrement` → "Frightened 3 → decrease to 2? [Accept] [Set to…] [Remove] [Dismiss]"
- `confirmSustained` → "Bless — sustained this turn? [Yes → Keep] [No → Remove]"
- `promptResolution` → "Persistent Fire 2d6 — take damage, flat check DC 15? [Passed → Remove] [Failed → Keep]"
- `reminder` → "Slowed 2 — loses 2 actions this turn. [OK]"

## 8. Amendments to Architecture Spec

### 8.1 Remove ADVANCE_TURN

The command vocabulary collapses ADVANCE_TURN into END_TURN. The architecture spec §16.3 should list: `SET_INITIATIVE_ORDER`, `REORDER_COMBATANT`, `END_TURN`, `DELAY`, `RESUME_FROM_DELAY`. ADVANCE_TURN is removed.

### 8.2 Add REVIVE Command

Added to §16.7 Combat State: `REVIVE` reverses `MARK_DEAD`.

### 8.3 Rename ADD_NOTE → SET_NOTE

`SET_NOTE` better communicates the semantics (idempotent set, not append).

### 8.4 Remove ADD_TO_INITIATIVE / REMOVE_FROM_INITIATIVE

These are covered by `REORDER_COMBATANT` (insert at position) and `REMOVE_COMBATANT`. Separate commands are unnecessary.

### 8.5 Automatic Reaction Reset

Reactions reset automatically at start-of-turn. This is unambiguous PF2e mechanical behavior and follows the same justification as hard clock expiry (arch spec A6). `RESET_REACTION` exists as a manual override only.

### 8.6 hp-reached-zero Event

When damage reduces HP to 0, the domain emits `hp-reached-zero` but does NOT auto-apply Dying or mark dead. This preserves GM authority — the GM decides whether this means Dying (PCs), instant death (NPCs), or narrative override.
